DaemonSet — One Pod Per Node

What it is, when to use it, and the dry-run-and-modify trick to deploy one without typing YAML from scratch.

One sentence definition

A DaemonSet is a controller that ensures every node (or every node matching a selector) runs exactly one copy of a pod. Add a node → a pod appears on it. Remove a node → its pod disappears.

The Mental Model

If a Deployment is "I want N pods scattered around the cluster", a DaemonSet is "I want one pod on every node — wherever there are nodes, there are pods." The scheduler isn't really making a choice; the controller is mirroring the node list.

analogy · the building inspector

Think of nodes as buildings and the DaemonSet as a building inspector. The rule is: every building must have one inspector on duty. Open a new building — an inspector is dispatched. Close a building — the inspector goes home. You don't choose how many inspectors total; the building count decides for you.

When to Reach for a DaemonSet

Anything that needs to be per-node, not per-app:

  • Log collectors — fluentd, fluent-bit, filebeat (read every node's logs).
  • Monitoring agents — node-exporter, datadog-agent (one metric source per node).
  • Networking — kube-proxy, CNI plugins (calico-node, cilium).
  • Storage drivers — CSI node plugins.
  • Security agents — falco, antivirus, runtime monitors.

DaemonSet vs Deployment

Deployment

You declare replicas: N. Scheduler decides where they land. May co-locate two replicas on one node.

"I want this many pods, somewhere."

DaemonSet

No replicas field. Pod count = node count (filtered by node selector / taints). One per node, max.

"I want one pod, on every node."

Anatomy of a DaemonSet

apiVersion: apps/v1                    # same group as Deployment
kind: DaemonSet
metadata:
  name: fluentd-elasticsearch
  namespace: kube-system
  labels:
    app: fluentd
spec:
  selector:                            # required, must match template labels
    matchLabels:
      app: fluentd
  template:                            # a normal Pod template
    metadata:
      labels:
        app: fluentd                   # MUST match selector above
    spec:
      tolerations:                     # run on control-plane too
        - key: node-role.kubernetes.io/control-plane
          operator: Exists
          effect: NoSchedule
      containers:
        - name: fluentd
          image: registry.k8s.io/fluentd-elasticsearch:1.20
          resources:
            requests:
              cpu: 100m
              memory: 200Mi
            limits:
              memory: 200Mi
      terminationGracePeriodSeconds: 30

Differences from a Deployment YAML

  • kind: DaemonSet instead of kind: Deployment.
  • No replicas: — count is dictated by node count.
  • No strategy: — DaemonSet uses updateStrategy instead (RollingUpdate / OnDelete).
  • Same apps/v1, same selector + template structure, same Pod spec.

Targeting Specific Nodes

You usually don't want a DaemonSet on literally every node. Two ways to filter:

Option 1 — nodeSelector on the template

spec:
  template:
    spec:
      nodeSelector:
        disktype: ssd                  # only nodes with this label

Option 2 — nodeAffinity for richer rules

spec:
  template:
    spec:
      affinity:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
              - matchExpressions:
                  - key: node-role.kubernetes.io/worker
                    operator: Exists

Option 3 — Tolerations for tainted nodes

Control-plane nodes are tainted by default with node-role.kubernetes.io/control-plane:NoSchedule. To run on them anyway (e.g. kube-proxy needs to), the DaemonSet must tolerate that taint.

The Imperative Trick (no kubectl create ds)

the gap

kubectl create does not support DaemonSet. There's no kubectl create ds. You'll get an error if you try.

The official workaround: generate a Deployment YAML, then surgically convert it.

Step 1 — Scaffold a Deployment

kubectl create deployment fluentd-elasticsearch \
  --image=registry.k8s.io/fluentd-elasticsearch:1.20 \
  -n kube-system \
  --dry-run=client -o yaml > ds.yaml

Step 2 — Edit four things

  1. Change kind: Deployment DaemonSet
  2. Delete the replicas: line
  3. Delete the strategy: {} line
  4. Delete the trailing status: {} line

Step 3 — Apply

kubectl apply -f ds.yaml

One-liner version (sed)

kubectl create deployment fluentd-elasticsearch \
  --image=registry.k8s.io/fluentd-elasticsearch:1.20 \
  -n kube-system --dry-run=client -o yaml \
  | sed -e 's/kind: Deployment/kind: DaemonSet/' \
        -e '/replicas:/d' \
        -e '/strategy:/d' \
        -e '/status:/,$d' \
  | kubectl apply -f -
✓ Why this works — A Deployment template and a DaemonSet template share the exact same selector + template structure. The only fields a Deployment has that a DaemonSet doesn't are replicas, strategy, and a few status fields. Strip those and you've got a valid DaemonSet.

Common Mistakes

✗ Mistake 1 — Running kubectl create ds my-app --image=...
That subcommand doesn't exist. kubectl exits with error: unknown command "ds". Use the dry-run trick.
✗ Mistake 2 — Setting replicas: on a DaemonSet.
The API server rejects it: strict decoding error: unknown field "spec.replicas". The number of pods is decided by the node count.
✗ Mistake 3 — selector.matchLabels not matching template.metadata.labels.
Same trap as Deployments. The DaemonSet will be rejected at create time.
✗ Mistake 4 — Expecting it to run on control-plane nodes by default.
It won't, unless you add a toleration for node-role.kubernetes.io/control-plane:NoSchedule. Things like kube-proxy and CNI agents tolerate everything precisely for this reason.
✗ Mistake 5 — Typo in the image registry.
egistry.k8s.io should be registry.k8s.io. Pod stays in ImagePullBackOff. Always check kubectl describe pod for image errors.
✗ Mistake 6 — Forgetting kind: DaemonSet after using the dry-run trick.
If you only delete replicas but forget to flip the kind, you've just built a Deployment with one pod. It schedules to one node, not all.

Update Strategies

DaemonSet pods get rolled differently from Deployments. Two options:

StrategyBehaviour
RollingUpdate (default) Kill old pods one by one (respecting maxUnavailable), spin up new ones. Same node, same time.
OnDelete Don't auto-replace anything. New pods only appear when you delete the old ones manually. Useful for very disruptive updates. 
spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1                # or a percentage like "10%"

Useful Commands

# List DaemonSets in a namespace
kubectl get ds -n kube-system

# Wide view — desired vs current vs ready
kubectl get ds fluentd-elasticsearch -n kube-system

# See the pods it created (one per node)
kubectl get pods -n kube-system -l app=fluentd -o wide

# Describe (good for troubleshooting scheduling)
kubectl describe ds fluentd-elasticsearch -n kube-system

# Roll out a new image
kubectl set image ds/fluentd-elasticsearch \
  fluentd=registry.k8s.io/fluentd-elasticsearch:1.21 -n kube-system

# Watch the rollout
kubectl rollout status ds/fluentd-elasticsearch -n kube-system

# Roll back if something breaks
kubectl rollout undo ds/fluentd-elasticsearch -n kube-system

# Delete the DaemonSet (and all its pods)
kubectl delete ds fluentd-elasticsearch -n kube-system

Reading the STATUS Columns

NAME           DESIRED  CURRENT  READY  UP-TO-DATE  AVAILABLE  NODE SELECTOR
fluentd-es     3        3        3      3           3          <none>
  • DESIRED — how many nodes the DaemonSet wants a pod on (after node selectors/taints are applied).
  • CURRENT — pods that exist.
  • READY — pods passing readiness probes.
  • UP-TO-DATE — pods running the latest spec.
  • AVAILABLE — pods ready for at least minReadySeconds.

If DESIRED ≠ CURRENT, scheduling is being blocked — usually a missing toleration, a nodeSelector that matches no nodes, or insufficient resources.

Imperative Quick Commands for CKA Exam

There is no kubectl create ds. Generate Deployment YAML, then convert it.

# Generate starter YAML
kubectl create deployment fluentd \
  --image=registry.k8s.io/fluentd-elasticsearch:1.20 \
  --dry-run=client -o yaml > ds.yaml

# Edit ds.yaml:
# kind: Deployment -> DaemonSet
# delete replicas, strategy, and status
vi ds.yaml
kubectl apply -f ds.yaml

# Verify one pod per node
kubectl get ds
kubectl get pods -l app=fluentd -o wide
