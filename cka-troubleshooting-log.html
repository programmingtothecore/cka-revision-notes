Troubleshooting Log — PriorityClass Pod YAML

A super-brief recap of what went wrong, what questions came up, and the clean fix path.

Topic

Creating a critical Pod with priorityClassName: high-priority, CPU/memory requests, and clean YAML for CKA.

What We Did

  2. Created an Imperative Commands Cheat Sheet page for CKA speed commands and expected output.
  3. Fixed cheat sheet readability by wrapping long commands instead of forcing horizontal scroll.

  5. Debugged why the edited YAML still failed.
  6. Added a Multiple Schedulers page after hitting the spec.scheduler field-name error.

Questions Asked

QuestionShort answer
Is there an imperative command to create a Pod with PriorityClass? Yes, use kubectl run ... --overrides='{"spec":{"priorityClassName":"high-priority"}}'.
Why is exported YAML painful? kubectl get pod -o yaml includes server-generated fields that are noisy or unsafe to reapply.
How do I quit vim without saving? Esc, then :q!, then Enter.
How do I delete a whole block in vim? Use V for visual line mode, select lines, then press d. Or use 5dd for 5 lines.
How do I undo in vim? Esc, then u. Redo is Ctrl+r.
Do I only delete annotations? No. For clean YAML, also remove generated metadata like creationTimestamp, resourceVersion, uid, and noisy runtime spec fields.
Why did my multiple-scheduler Pod fail with unknown field "spec.scheduler"? The field is spec.schedulerName, not spec.scheduler.

The Actual YAML Bug

typo

The file had this:

piVersion: v1

It must be:

apiVersion: v1

Clean Final YAML

apiVersion: v1
kind: Pod
metadata:
  name: critical-app
  namespace: default
spec:
  priorityClassName: high-priority
  containers:
  - name: critical-container
    image: nginx
    resources:
      limits:
        memory: 3Gi
      requests:
        cpu: "1"
        memory: 3Gi

Apply Flow

# Recreate because most Pod spec fields are immutable
kubectl delete pod critical-app --ignore-not-found

# Apply clean YAML
kubectl apply -f crtapp.yaml

# Check status and scheduling events
kubectl get pod critical-app -o wide
kubectl describe pod critical-app

Multiple Scheduler Issue

field name

The YAML used this invalid field:

spec:
  scheduler: my-scheduler

Kubernetes rejected it with:

strict decoding error: unknown field "spec.scheduler"

The correct field is:

spec:
  schedulerName: my-scheduler

Clean YAML

apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  schedulerName: my-scheduler
  containers:
  - name: nginx
    image: nginx

Imperative Quick Commands for CKA Exam

# Generate clean starter YAML
kubectl run critical-app \
  --image=nginx \
  --dry-run=client -o yaml > crtapp.yaml

# Existing Pod export, if needed
kubectl get pod critical-app \
  -o yaml > crtapp.yaml

# Pod with PriorityClass in one command
kubectl run critical-app \
  --image=nginx \
  --overrides='{"spec":{"priorityClassName":"high-priority"}}'

# Pod using custom scheduler
kubectl run nginx \
  --image=nginx \
  --overrides='{"spec":{"schedulerName":"my-scheduler"}}'
The Mantra "Generate clean YAML when possible. Exported YAML needs cleanup. Check apiVersion first."
