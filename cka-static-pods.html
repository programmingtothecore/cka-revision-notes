Static Pods — Kubelet-Managed Pods

How pods can run without the scheduler, why control-plane components use them, and how to inspect, edit, and recover them in the exam.

One sentence definition

A Static Pod is a pod defined by a manifest file on a node, watched directly by the kubelet, and restarted by that kubelet without needing a Deployment, ReplicaSet, DaemonSet, or scheduler.

The Mental Model

Most pods are born through the API server: you create YAML, the scheduler picks a node, and the kubelet on that node runs the container. A Static Pod skips that path. The kubelet reads a local file, creates the pod locally, and keeps it alive.

analogy · the local instruction sheet

Imagine every node has a clipboard taped to its door. If a pod spec appears on that clipboard, the kubelet says, "This is my own instruction; I will run it here." No scheduler interview. No controller deciding replicas. Just the node following its local sheet.

Why Static Pods Exist

Static Pods are perfect for bootstrapping the control plane because they do not need the control plane to already be healthy.

  • API server can be started by kubelet even before the Kubernetes API is usable.
  • Controller manager and scheduler can run as local manifests on control-plane nodes.
  • etcd often runs the same way in kubeadm clusters.
  • If a control-plane component crashes, the kubelet restarts it from the manifest.

Static Pod vs Normal Pod

Normal Pod

Created through the API server. Usually managed by a controller. Scheduler chooses a node.

"Kubernetes cluster, please run this pod somewhere."

Static Pod

Created from a file on one node. Managed by the kubelet on that node. Scheduler is bypassed.

"This node, run this pod here."

The Control Loop

  1. Kubelet starts on a node.
  2. Kubelet reads its configured static pod path, usually /etc/kubernetes/manifests.
  3. Every YAML file in that directory becomes a pod on that node.
  4. If the file changes, kubelet updates/recreates the pod.
  5. If the file is deleted or moved away, kubelet stops the pod.
✓ Exam shortcut — On kubeadm clusters, the static pod directory is usually /etc/kubernetes/manifests. That is where you will find kube-apiserver.yaml, kube-controller-manager.yaml, kube-scheduler.yaml, and often etcd.yaml.

The Mirror Pod

Static Pods are local, but they still appear when you run kubectl get pods. That visible API object is called a mirror pod.

ThingWhere it livesWhat controls it
Static pod manifest Node filesystem, like /etc/kubernetes/manifests Kubelet
Real running pod The node where the manifest exists Kubelet / container runtime
Mirror pod API server view, visible through kubectl Read-only reflection of the static pod
important

You cannot manage a Static Pod by deleting the mirror pod from the API server. The kubelet sees the local manifest still exists and recreates the mirror pod. To actually remove it, delete or move the manifest file from the node.

A Minimal Static Pod

apiVersion: v1
kind: Pod
metadata:
  name: static-nginx
  labels:
    app: static-nginx
spec:
  containers:
    - name: nginx
      image: nginx:1.25
      ports:
        - containerPort: 80

Place that file on the node in the kubelet's static pod path:

sudo cp static-nginx.yaml /etc/kubernetes/manifests/

The kubelet notices the new file and creates the pod on that same node.

How to Find the Static Pod Path

Do not guess if the cluster is unusual. Read the kubelet config.

# Common kubeadm location
sudo grep -n "staticPodPath" /var/lib/kubelet/config.yaml

# Typical result
staticPodPath: /etc/kubernetes/manifests

You may also see a kubelet flag called --pod-manifest-path in older setups.

# Check kubelet process flags
ps -ef | grep kubelet

# Or inspect the systemd unit/drop-ins
systemctl cat kubelet

Editing Control Plane Static Pods

In a kubeadm cluster, control-plane components are Static Pods. Editing their manifest file is how you change their command flags, volumes, images, or certificates.

# Control-plane manifests
ls /etc/kubernetes/manifests

# Example edit
sudo vi /etc/kubernetes/manifests/kube-apiserver.yaml
be careful

When you save the file, kubelet restarts that component. A bad edit can take down the API server. Make small changes, keep another shell open, and check the pod status quickly.

Useful Commands

# Show static/mirror pods in the kube-system namespace
kubectl get pods -n kube-system -o wide

# Control-plane static pods usually include the node name
kubectl get pods -n kube-system | grep kube-apiserver

# Describe a mirror pod
kubectl describe pod kube-apiserver-controlplane -n kube-system

# Check kubelet logs if the static pod will not come up
journalctl -u kubelet -f

# Check the local manifest directory
sudo ls -l /etc/kubernetes/manifests

# Temporarily stop a static pod by moving the manifest out
sudo mv /etc/kubernetes/manifests/static-nginx.yaml /tmp/

# Restore it by moving the manifest back
sudo mv /tmp/static-nginx.yaml /etc/kubernetes/manifests/

Static Pods Cannot Use Everything

A Static Pod uses normal kind: Pod YAML, but there are important limits because no controller owns it.

FeatureCan a Static Pod use it?Why
replicas No Pods do not have replicas. Only controllers like Deployments do.
Scheduling to another node No The pod runs on the node where the manifest file exists.
ConfigMaps / Secrets Usually yes If the API server is reachable; risky for bootstrapping components that must start before API health.
ServiceAccount tokens Not like normal API-created pods Static Pods are not created by the usual admission path.

Common Mistakes

✗ Mistake 1 — Trying to create a Static Pod with kubectl apply.
That creates a normal API-server pod. A Static Pod is created by putting a pod manifest on the node where kubelet watches it.
✗ Mistake 2 — Deleting the mirror pod and expecting it to stay gone.
The kubelet recreates it because the local manifest still exists. Remove the manifest file to stop the real pod.
✗ Mistake 3 — Looking for a Deployment or ReplicaSet owner.
Static Pods do not have a normal workload controller. The owner is effectively the kubelet.
✗ Mistake 4 — Forgetting that the scheduler is bypassed.
nodeSelector, affinity, and taints are not the main placement story here. The manifest's filesystem location decides the node.
✗ Mistake 5 — Breaking kube-apiserver.yaml without a rollback path.
If the API server disappears, use node access and kubelet logs. Fix the local file; kubectl may not be available while the API server is down.

Quick-Recall Drill

QuestionAnswer
Who manages a Static Pod?The kubelet on that node.
Does the scheduler place it?No. It runs where the manifest file exists.
Common manifest path?/etc/kubernetes/manifests
How do you stop one?Delete or move the manifest file from the watched directory.
Why does it show in kubectl get pods?The kubelet creates a mirror pod in the API server.
Can you edit the mirror pod directly?No. Edit the local manifest file.

Imperative Quick Commands for CKA Exam

Static Pods are not created with kubectl apply. Create or edit the manifest file on the node in the kubelet's watched directory.

# Find the static pod manifest path
sudo grep -n "staticPodPath" /var/lib/kubelet/config.yaml

# Generate a Pod manifest quickly
kubectl run static-nginx --image=nginx --dry-run=client -o yaml > static-nginx.yaml

# Move it into the static pod directory on that node
sudo mv static-nginx.yaml /etc/kubernetes/manifests/

# Stop the static pod
sudo mv /etc/kubernetes/manifests/static-nginx.yaml /tmp/

# Watch kubelet while it reacts
journalctl -u kubelet -f
