How to Guess the apiVersion

Three reliable shortcuts — a mental cheat sheet, a structural rule, and the command that ends all guessing.

The TL;DR

Pod-era basics → v1. Workloads that manage pods → apps/v1. Subsystems → their own group like networking.k8s.io/v1. When in doubt → kubectl explain <kind>.

Memorise this table. It covers about 90% of what the CKA throws at you.

2 · The Structural Rule

If you can't recall a row, derive it from the kind. Three patterns explain almost everything:

pattern · pod-era primitives

"Old / fundamental" → bare v1

The original Kubernetes objects — the things invented alongside the Pod itself — live in the core API group, which has no group name. They show up in YAML as just apiVersion: v1.

Tell: can you imagine running this on day one of Kubernetes? Pod, Service, ConfigMap, Secret, Namespace, PV/PVC, ServiceAccount → all v1.

pattern · workloads

"Manages or scales pods" → apps/v1

Anything whose job is to run pods on your behalf sits in the apps group: Deployment, StatefulSet, DaemonSet, ReplicaSet. They were extracted out of core when controllers grew up.

Tell: if removing the object would also remove its pods, it's probably apps/v1.

pattern · subsystems

"Whole topic" → its own named group

Bigger areas of Kubernetes get their own group whose name reads like the topic. The version suffix is almost always /v1.

  • Networking → networking.k8s.io/v1
  • RBAC → rbac.authorization.k8s.io/v1
  • Storage → storage.k8s.io/v1
  • Certificates → certificates.k8s.io/v1
  • Admission webhooks → admissionregistration.k8s.io/v1
  • CRDs → apiextensions.k8s.io/v1

Tell: the topic name is right there in the group name.

pattern · the historical outliers

Short group names without .k8s.io

A handful of groups predate the .k8s.io naming convention and never got renamed: apps, batch, policy, autoscaling. Don't overthink it — just memorise these four short names.

3 · Stop Guessing — Ask the Cluster

In the exam, you have a live cluster. Use it.

# The fastest way: prints apiVersion on line 2
kubectl explain pod         | head -2
kubectl explain deployment  | head -2
kubectl explain ingress     | head -2
kubectl explain hpa         | head -2

# Full list — every kind, its apiVersion, namespaced or not
kubectl api-resources

# All API versions enabled on this cluster
kubectl api-versions

# Best of all: scaffold a manifest with the right apiVersion baked in
kubectl create deployment web --image=nginx --dry-run=client -o yaml
kubectl run nginx --image=nginx --dry-run=client -o yaml
kubectl create job hello --image=busybox --dry-run=client -o yaml -- echo hi
kubectl create cronjob hi --image=busybox --schedule='*/5 * * * *' \
        --dry-run=client -o yaml -- echo hi
kubectl create role dev --verb=get,list --resource=pods --dry-run=client -o yaml
✓ Exam habit — never type a manifest from scratch. Always start with kubectl create … --dry-run=client -o yaml > file.yaml, then edit. The right apiVersion is already there.

Version Suffix — Quick Notes

The bit after the slash (/v1, /v1beta1, etc.) tells you the maturity:

SuffixMeaningUse in production?
v1Stable. API guarantees apply.Yes
v2Stable, second iteration of an API (e.g., HPA).Yes
v1beta1 / v1beta2Beta — feature complete but may still change.Mostly OK
v1alpha1Alpha — experimental. Off by default. Can break.No

The Three Most-Confused Pairs

✗ Pitfall 1 — Writing apiVersion: apps/v1 for a Pod or Service.
Pods and Services are core — just v1. The apps/ prefix is only for workload controllers.
✗ Pitfall 2 — Using extensions/v1beta1 for Ingress or Deployment.
That group was deprecated and removed in 1.16+. Modern: Deployment → apps/v1, Ingress → networking.k8s.io/v1.
✗ Pitfall 3 — Writing autoscaling/v1 for an HPA with custom metrics.
v1 only supports CPU. Use autoscaling/v2 for memory, custom, or external metrics.

One-Glance Decision Tree

What kind am I writing?
│
├─ Pod, Service, ConfigMap, Secret, Namespace, PV/PVC, SA?
│     → apiVersion: v1
│
├─ Deployment, StatefulSet, DaemonSet, ReplicaSet?
│     → apiVersion: apps/v1
│
├─ Job, CronJob?
│     → apiVersion: batch/v1
│
├─ Ingress, NetworkPolicy?
│     → apiVersion: networking.k8s.io/v1
│
├─ Role / Binding (any flavour)?
│     → apiVersion: rbac.authorization.k8s.io/v1
│
├─ HPA?
│     → apiVersion: autoscaling/v2
│
└─ Anything else?
      → kubectl explain <kind> | head -2

Imperative Quick Commands for CKA Exam

# Discover apiVersion/kind from the cluster
kubectl api-resources | grep -i deployment
kubectl api-resources | grep -i priority

# Ask Kubernetes for the exact field tree
kubectl explain deployment
kubectl explain deployment.spec.template.spec.containers
kubectl explain priorityclass

# Generate YAML and inspect apiVersion/kind
kubectl create deployment nginx --image=nginx --dry-run=client -o yaml
kubectl run nginx --image=nginx --dry-run=client -o yaml
kubectl create job hello --image=busybox --dry-run=client -o yaml -- echo hi
