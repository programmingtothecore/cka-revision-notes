Asymmetric Encryption, Signatures & Certificates — Plain English

A visual, no-hand-waving map of public/private keys, signatures, hashes, certificates, trust stores, CA chains, mTLS, and what this still does not solve.

One sentence definition

Asymmetric crypto solves secure communication between strangers by separating public tools from private secrets, then using signatures and certificates to prove identity and integrity.

The Starting Problem

Symmetric encryption uses one key to both encrypt and decrypt. That is fine if you and I already share that key. But if we have never met, how do I send you the key in the first place? Email it? Anyone watching the wire grabs it. This is the key distribution problem.

Symmetric Problem

Fast encryption, but both sides need the same secret key first.

Asymmetric Trick

Public key is shareable. Private key never leaves your machine.

Padlock Model

Anyone can lock a box with your public padlock. Only your private key opens it.

Public Key / Private Key

Asymmetric encryption uses two mathematically linked keys. The public key can be shouted from the rooftops. The private key is guarded and never leaves your machine. Anything encrypted with the public key can only be decrypted with the matching private key. Even the public key itself cannot reverse what it just did.

The math rests on one-way problems. RSA uses the fact that multiplying two huge prime numbers is easy, but factoring the result back into those primes is infeasible. Elliptic curve crypto uses point multiplication on a curve. You do not need the math for the mental model; you need the idea of easy one way, infeasible backward.

The Four Operations

This is where many people get surprised. A private key does two jobs, not one. A public key also does two jobs.

OperationKey UsedPurpose
Encrypt to BB's public keyOnly B can read it.
Decrypt for BB's private keyB opens what was encrypted to B.
Sign as AA's private keyA proves this message came from A.
Verify AA's public keyAnyone checks A's signature.

Encryption Alone Is Not Enough

A: Yo I wanna send you a message but they're spying on us.

B: Send it using my public key. I am the only one who can decrypt it.

A encrypts message with B's public key and sends it.

B: I decrypt with my private key and read it.

Clean, but incomplete. What if a hacker also encrypts garbage with B's public key and sends it? B can read it, but B cannot tell whether A sent it. Encryption gives confidentiality, not authenticity.

Signatures Fix Authenticity

A signature is not a key. It is a number derived from two inputs: the hash of the message and the sender's private key. First, run the message through a hash function like SHA-256. That produces a fixed-size fingerprint. Change one bit of the message and the hash changes wildly. Then sign that hash with the private key.

Hash the message. Message becomes a fixed-size fingerprint.
Sign the hash. Private key creates a signature bound to that hash.
Send message + signature. You send the stamp impression, not the stamp.
Receiver verifies. Public key checks whether the signature matches the message.

Verification, No Magic

This is the part explanations often gloss over. Verification is an explicit comparison.

StepReceiver DoesResult
1Hash the received message.Hash X
2Use sender's public key to recover/check the hash from the signature.Hash Y
3Compare X and Y.Equal = authentic and untampered. Not equal = reject.
important

The public key stores nothing. It has no memory and no database of old messages. It simply enables a math check that runs fresh every time.

Full Message Flow

A: Yo I wanna send you a message but they're spying.

B: You got my public key right? I've had it published forever.

A: Yep. Here's mine too.

A writes message, hashes it, signs the hash with A's private key, bundles message + signature, encrypts the bundle with B's public key, and sends it.

B: I got five bundles. Four fake, one real. I decrypt each with my private key, hash the plaintext, verify signature with A's public key, and compare hashes.

B: Fake bundles fail. Real bundle matches. Got your message, signature checked out clean.

What This Achieves

Confidentiality

Encryption means only B's private key can read the bundle.

Authenticity

Signature means B can tell the real sender from attacker noise.

Integrity

Hash comparison means tampering breaks verification.

If a Signature Is Stolen

A stolen signature is not like a stolen private key. It is bound to one specific message. You cannot graft it onto another message because the hash will not match. You cannot derive the private key from it. You cannot forge new signatures from it.

Stolen ThingAttacker Can DoSeverity
One signatureMaybe replay that exact signed message.Limited
Many signaturesStill cannot derive the private key if crypto is implemented correctly.Usually limited
Private keySign anything forever until revoked/rotated.Game over

The replay risk is real: an attacker may re-send your authentic signed message later. Real protocols include a nonce, timestamp, or sequence number inside the signed message so duplicates and stale messages can be rejected.

The Giant Remaining Hole

How does A know B's public key is actually B's? If they have never talked before, an attacker in the middle can swap in their own public key. A encrypts to the attacker, attacker decrypts, reads, re-encrypts to B, and forwards. The crypto math is flawless, but it protects the wrong key. This is the man-in-the-middle problem.

Certificates Solve Identity Binding

A certificate is a signed document that says: "I, a Certificate Authority, vouch that this public key belongs to this identity."

Subject: bank.com
Subject's Public Key: the actual public key
Issuer: DigiCert
Valid From: 2026-01-01
Valid Until: 2027-01-01
Signature: signed by DigiCert's private key

The signature on the cert is created like any other signature. The CA hashes the cert contents and signs the hash with the CA's private key. Same signing mechanism, but now applied to identity documents.

Trust Store Logic

How do you know the CA's public key is real? Your OS/browser ships with trusted CA public keys pre-installed in a trust store. Windows has one, macOS has one, Linux distros have one, browsers have one. When you installed your OS, you implicitly trusted those CAs.

A receives B's certificate.
A checks the issuer: DigiCert.
A finds DigiCert's public key in the OS/browser trust store.
A verifies the cert signature with DigiCert's public key.
A checks validity dates and subject/SAN matches the domain.
Only now does A trust the public key inside B's certificate.

Certificate Dialogue

A: I want to talk to B, but I have never talked to him. No idea what his real public key is.

B: Here is my certificate. It has my public key inside, plus a signature from DigiCert.

A: First, do I trust DigiCert? Checking trust store... yes.

A: Now I hash the cert contents myself, recover/check the hash from the signature using DigiCert's public key, and compare.

A: Hashes match. Cert is in date. Subject says bank.com. Now I trust B's public key.

precise meaning

The cert signature proves exactly one thing: whoever holds the private key matching DigiCert's public key signed this exact cert content. Trust comes from a separate judgment: DigiCert is in the trust store.

Certificate Chains

Real deployments usually use chains. Root CAs keep their private keys offline in vaults. They sign intermediate CAs. Intermediates sign leaf certificates for websites. Browser walks the chain: leaf → intermediate → root in trust store. Any broken link rejects the connection.

LayerRoleWhy
Root CATrust anchor in OS/browser store.Kept offline because compromise is catastrophic.
Intermediate CASigns day-to-day certificates.Can be revoked/replaced if compromised.
Leaf certCertificate for bank.com / service endpoint.Contains the actual server public key.

Other Certificate Fields

  • Subject Alternative Names: domains the cert covers, like bank.com, www.bank.com, api.bank.com.
  • Key Usage: what the cert is allowed to do, such as server auth, client auth, code signing.
  • Revocation: CRL and OCSP let clients check whether a cert was revoked before expiry.
  • Short-lived certs: Let's Encrypt's 90-day certs reduce damage if revocation fails.

Self-Signed Certs

A self-signed cert is a valid X.509 cert where you sign your own cert with your own private key. No public browser trusts it by default because it does not chain to a trusted root. It is common for internal services, dev environments, corporate proxies, and private PKI where you control the trust store.

Fake CA Question

Can you create a CA called "DigiCerts" and sign certificates? Technically yes. The certificates can be cryptographically valid. But browsers reject them because your fake CA public key is not in their trust stores. Trust stores match public keys, not names.

Getting into real trust stores requires audits, HSM-backed key storage, documented key ceremonies, published policies, and ongoing public scrutiny. CAs that misbehave can be distrusted, as happened with DigiNotar, Symantec, and TrustCor. Certificate Transparency adds another layer by logging public CA-issued certs in public append-only logs.

mTLS Full Flow

In normal HTTPS, usually only the server has a certificate. In mTLS, both sides have certificates, common in service-to-service auth in Kubernetes, service meshes, and enterprise B2B systems.

B sends certificate. A verifies it against A's trust store and extracts B's public key.
A sends certificate. B verifies it against B's trust store and extracts A's public key.
A writes message, hashes it, signs with A's private key.
A bundles message + signature + nonce + timestamp.
A encrypts the bundle to B.
B decrypts, verifies A's signature, checks nonce/timestamp, and accepts only the valid message.

Still Not Enough

This foundation explains the primitives, but real TLS does more.

Remaining ProblemReal-World Fix
Asymmetric crypto is slow.Use it to bootstrap a fast symmetric key, then use AES/ChaCha20.
No forward secrecy if long-term private key leaks.Use ephemeral Diffie-Hellman, as in TLS 1.3.
Replay attacks.Use nonces, timestamps, and sequence numbers.
Endpoint compromise.Use key vaults, HSMs, least privilege, endpoint hardening.
Future quantum attacks.Migrate toward post-quantum cryptography.

Imperative Quick Commands for CKA Exam

This topic is more conceptual than kubectl-heavy, but these commands help when dealing with Kubernetes TLS material and cert debugging.

# Inspect a certificate
openssl x509 -in tls.crt -text -noout

# Show issuer, subject, and dates
openssl x509 -in tls.crt -noout -issuer -subject -dates

# Check Kubernetes cert expiry on kubeadm clusters
kubeadm certs check-expiration

# Inspect a TLS secret
kubectl get secret my-tls -o yaml

# Decode cert from a TLS secret
kubectl get secret my-tls \
  -o jsonpath='{.data.tls\.crt}' | base64 -d | openssl x509 -text -noout
