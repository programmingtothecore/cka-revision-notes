The Kubernetes Scheduling Story

Selectors, Affinity & Tolerations made memorable — a job-fair in three acts.

The Cast

Imagine a job fair in a city

  • Companies — Controllers (Deployment, Service, NetworkPolicy)
  • Workers — Pods
  • Houses — Nodes

Act 1 — Companies hire workers (Selectors)

act 1 · selectors

Companies post job ads. Workers wear T-shirts with labels like app=red.

Simple ad — matchLabels

"Hiring anyone wearing app=red."

Picky ad — matchExpressions

"Hiring anyone whose shirt is In [red, blue]."

Set-based operators: In, NotIn, Exists, DoesNotExist.

Service is the lazy company — it skips the matchLabels wrapper and writes app: red directly. Flat.

Act 2 — Workers pick houses (Affinity)

act 2 · affinity

Hired workers need somewhere to live. Houses have labels too: color=blue, disktype=ssd.

Lazy worker — nodeSelector

"I'll live in any house labeled disktype=ssd."

Flat key: value map. Done.

Picky worker — nodeAffinity

"I require color In [blue] and kernel-version Gt 5."

Affinity is the richest language — it gets the bonus operators Gt and Lt on top of the four set operators.

Act 3 — Some houses are haunted (Taints & Tolerations)

act 3 · taints & tolerations

A few houses are tainted — haunted by ghosts. The owner shouts:

"color=blue:NoSchedule — stay away unless you can handle me!"

A worker who isn't scared writes a toleration:

"I tolerate color Equal blue, effect NoSchedule."

Tolerations are the weird cousin:

  • Only two operators: Equal, Exists
  • Singular value (not values!)
  • Don't attract the pod to the node — just allow it

A toleration is a permission slip, not a GPS.

Three-sentence summary
  1. Selectors = companies hiring workers by shirt label
  2. Affinity = workers choosing houses by house label
  3. Tolerations = workers willing to live in haunted houses

The One Table That Matters

Selectors nodeAffinity Tolerations
Operators In, NotIn, Exists, DoesNotExist + Gt, Lt Equal, Exists
Values field values (list) values (list) value (singular)
Lives on Controller Pod Pod

The Three Mistakes That Will Bite You

✗ Mistake 1 — Using Equal in affinity.
It's In (with a values list). Affinity has no Equal.
✗ Mistake 2 — Using value in affinity.
It's values (a list) for affinity/selectors. Only tolerations use singular value.
✗ Mistake 3 — Putting affinity at spec.affinity in a Deployment.
It's spec.template.spec.affinity — affinity is a pod concern, not a deployment concern.

Quick YAML Reference

Service selector — flat

kind: Service
spec:
  selector:
    app: red

Deployment matchLabels

spec:
  selector:
    matchLabels:
      app: red
  template:
    metadata:
      labels:
        app: red    # MUST match selector

matchExpressions

selector:
  matchExpressions:
    - key: app
      operator: In
      values:
        - red
        - blue

nodeSelector (simple)

spec:                # pod spec
  nodeSelector:
    disktype: ssd

nodeAffinity (rich)

spec:                # pod spec
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: color
                operator: In
                values:
                  - blue

Tolerations

spec:                # pod spec
  tolerations:
    - key: color
      operator: Equal
      value: blue          # singular!
      effect: NoSchedule

Required vs Preferred (Affinity only)

VariantHard or Soft?Has weight?
requiredDuringSchedulingIgnoredDuringExecutionHard rule — won't schedule if no matchNo
preferredDuringSchedulingIgnoredDuringExecutionSoft preference — best-effortYes (1–100)

Toleration Effects

EffectMeaning
NoScheduleNew pods won't schedule unless they tolerate
PreferNoScheduleSoft version — scheduler tries to avoid
NoExecuteExisting pods evicted unless they tolerate (use tolerationSeconds for grace period)

Imperative Quick Commands for CKA Exam

# Pod with nodeSelector
kubectl run nginx --image=nginx \
  --overrides='{"spec":{"nodeSelector":{"disktype":"ssd"}}}'

# Pod with toleration
kubectl run nginx --image=nginx \
  --overrides='{"spec":{"tolerations":[{"key":"color","operator":"Equal","value":"blue","effect":"NoSchedule"}]}}'

# Generate YAML for affinity, then edit
kubectl run nginx --image=nginx --dry-run=client -o yaml > pod-affinity.yaml
vi pod-affinity.yaml
kubectl apply -f pod-affinity.yaml

# Confirm placement and events
kubectl get pod nginx -o wide
kubectl describe pod nginx
