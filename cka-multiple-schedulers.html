Multiple Schedulers — Custom Pod Placement

Why clusters can run more than one scheduler, how pods choose a scheduler, and the exact YAML field that matters in the exam.

One sentence definition

Multiple schedulers let different pods be scheduled by different scheduler processes; a pod opts into a non-default scheduler with spec.schedulerName.

Purpose

The default scheduler is called default-scheduler. In some clusters, teams run an extra scheduler with custom scheduling logic. Pods that need that logic set schedulerName; normal pods continue using the default scheduler.

  • Custom placement rules beyond normal affinity, taints, and resources.
  • Special workloads such as batch, GPU, research, or internal platform workloads.
  • Experimentation with scheduling behavior without replacing the default scheduler.
  • Isolation of scheduling decisions for a subset of pods.
mental model

Think of the scheduler as the person assigning rooms. Most guests go to the default desk. Special guests can write schedulerName: my-scheduler and wait for a different desk to assign their room.

The Pod Field

This is the main CKA field:

spec:
  schedulerName: my-scheduler
exact trap

This is wrong and causes strict decoding error: unknown field "spec.scheduler":

spec:
  scheduler: my-scheduler

Pod YAML

apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  schedulerName: my-scheduler
  containers:
  - name: nginx
    image: nginx

How a Custom Scheduler Runs

In a kubeadm-style cluster, a scheduler can run as a normal pod, Deployment, or static pod. The important part is that its command uses a distinct scheduler name.

apiVersion: v1
kind: Pod
metadata:
  name: my-scheduler
  namespace: kube-system
spec:
  containers:
  - name: kube-scheduler
    image: registry.k8s.io/kube-scheduler:v1.30.0
    command:
    - kube-scheduler
    - --scheduler-name=my-scheduler
    - --leader-elect=false
✓ CKA focus — You are more likely to be asked to create a pod that uses an existing scheduler than to build a production-grade scheduler deployment from scratch.

Imperative Quick Commands for CKA Exam

# Generate clean Pod YAML
kubectl run nginx \
  --image=nginx \
  --dry-run=client -o yaml > nginx-pod.yaml

# Edit and add:
# spec:
#   schedulerName: my-scheduler
vi nginx-pod.yaml
kubectl apply -f nginx-pod.yaml

# One-command version with overrides
kubectl run nginx \
  --image=nginx \
  --overrides='{"spec":{"schedulerName":"my-scheduler"}}'

# Verify schedulerName
kubectl get pod nginx \
  -o jsonpath='{.spec.schedulerName}{"\n"}'

Troubleshooting

✗ Mistake 1 — Using spec.scheduler.
Correct field is spec.schedulerName.
✗ Mistake 2 — Using a scheduler name that is not running.
The pod can stay Pending because no scheduler is watching that name.
✗ Mistake 3 — Thinking schedulerName replaces node placement rules.
The scheduler still must satisfy resources, node selectors, affinity, taints, and tolerations.

Useful Commands

# See schedulerName on a pod
kubectl get pod nginx -o yaml | grep schedulerName

# Describe pod events
kubectl describe pod nginx

# Find scheduler pods
kubectl get pods -n kube-system | grep scheduler

# Check default scheduler static pod manifest
sudo ls /etc/kubernetes/manifests
sudo grep -n "scheduler-name" /etc/kubernetes/manifests/kube-scheduler.yaml
