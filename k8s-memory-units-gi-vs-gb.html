Memory Units — Gi vs GB

Why 1Gi isn't 1GB — and when that 7% actually bites you.

The Core Distinction

Kubernetes resource specs support two suffix families: decimal (powers of 1000) and binary (powers of 1024). The lowercase i is the giveaway — it means binary.

Two definitions, side by side

1 GB = 1,000,000,000 bytes  (10⁹, decimal)
1 Gi = 1,073,741,824 bytes  (2³⁰, binary)
Difference: ~7.4%

For mental math during exams or planning, 1Gi ≈ 1GB is close enough. The 7% only matters when you're packing pods tightly onto nodes.

Pod Spec Example

resources:
  requests:
    memory: "512Mi"   # 512 × 1024 × 1024 bytes
    cpu: "500m"
  limits:
    memory: "1Gi"     # 1024 × 1024 × 1024 bytes

Why Binary?

Linux memory allocators work in powers of 2 — page sizes are 4 KiB, hugepages are 2 MiB, and so on. Using Mi/Gi reflects what the kernel actually allocates. That's why nearly every production manifest uses binary suffixes for memory.

⚠ Real-world gotcha — An Azure Standard_D4s_v5 node advertises 16 GB RAM — but that's only ~14.9 Gi. After kubelet and system reserves, you have less still. So 2 × 8Gi pods won't fit, even though "16 GB = 2 × 8 GB" looks correct on paper.

Pods stay Pending with Insufficient memory — and the math seems fine until you remember the unit mismatch.

What About Gb (lowercase b)?

Doesn't exist in Kubernetes. Gb means gigabit — a networking unit (e.g., 1 Gbps link). All Kubernetes resource specs deal in bytes, never bits.

Common Mistakes

✗ Mistake 1 — Writing memory as 500m thinking it's "500 mega".
That's 500 milli-bytes ≈ 0.5 bytes. Container will fail to start.
✓ Use: 500Mi or 500M.
✗ Mistake 2 — Mixing G and Gi across requests/limits.
Looks fine, schedules fine, but you've quietly given the pod 7% less (or more) than you intended.
✗ Mistake 3 — Assuming the node's advertised RAM is what's available.
kubelet, kube-proxy, OS reserves, and eviction thresholds eat into it. Always check kubectl describe node for Allocatable.

Imperative Quick Commands for CKA Exam

# Create a Pod with memory request/limit
kubectl run nginx --image=nginx \
  --requests='memory=128Mi,cpu=100m' \
  --limits='memory=256Mi,cpu=500m'

# Generate YAML to verify exact units
kubectl run nginx --image=nginx \
  --requests='memory=128Mi' \
  --limits='memory=1Gi' \
  --dry-run=client -o yaml > pod-resources.yaml

# Check what was applied
kubectl get pod nginx -o yaml | grep -A8 resources
