Pods & Controllers — The Basics Map

The quick relationship chart: Pod, ReplicationController, ReplicaSet, Deployment, DaemonSet, StatefulSet, Job, CronJob, and Service.

One sentence definition

A Pod is the smallest runnable unit; controllers are the objects that keep pods created, replaced, scaled, updated, or placed in special ways.

The Owner Chain

For normal app workloads, memorize this chain:

Deployment
  └── ReplicaSet
        └── Pod

You usually create the Deployment. It creates a ReplicaSet. The ReplicaSet creates the Pods. If a Pod dies, the ReplicaSet replaces it. If you change the Deployment template, the Deployment creates a new ReplicaSet and rolls traffic toward it.

mental model

Pod is the worker. ReplicaSet is the headcount checker. Deployment is the manager who handles updates and rollbacks.

The Main Objects

ObjectapiVersionWhat it doesCKA memory hook
Pod v1 Runs one or more containers together on one node. Smallest unit. Usually don't run apps with bare Pods.
ReplicationController v1 Old controller that keeps N matching pods alive. Legacy. Uses equality selectors only.
ReplicaSet apps/v1 Modern replacement for ReplicationController. Supports matchLabels and matchExpressions.
Deployment apps/v1 Manages ReplicaSets and rolling updates. Default choice for stateless apps.
DaemonSet apps/v1 Runs one pod on each matching node. Logs, metrics, CNI, node agents.
StatefulSet apps/v1 Runs pods with stable names, order, and storage identity. web-0, web-1, stable PVCs.
Job batch/v1 Runs pods until a task completes successfully. Finite work, not forever services.
CronJob batch/v1 Creates Jobs on a schedule. Backup every night, report every hour.
Service v1 Gives stable networking to matching pods. Does not create pods. Selects pods by labels.

ReplicationController vs ReplicaSet

They both try to keep a desired number of pods running. The difference is selector power and age.

ReplicationControllerReplicaSet
Modern? No, older object Yes, modern object
apiVersion v1 apps/v1
Selector Simple equality selector matchLabels and matchExpressions
Use directly? Rarely Rarely; Deployment usually owns it
✓ Exam shortcut — If asked to create a scalable stateless app, use kubectl create deployment. If asked specifically about older replication behavior, know that ReplicationController is the legacy cousin of ReplicaSet.

Selector Relationship

Controllers do not own pods by name. They find pods by labels.

spec:
  selector:
    matchLabels:
      app: nginx                 # controller looks for this
  template:
    metadata:
      labels:
        app: nginx               # pods must wear this
✗ Common trap — If selector.matchLabels does not match template.metadata.labels, a Deployment, ReplicaSet, or DaemonSet will fail validation or manage the wrong pods.

When to Use What

NeedUseWhy
Run one quick test containerPodFastest minimal object.
Run a stateless web appDeploymentScaling, rollout, rollback.
Run one agent per nodeDaemonSetPod count follows node count.
Run database-like pods with identityStatefulSetStable names and storage.

Mini YAML Shapes

Bare Pod

apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
    - name: nginx
      image: nginx

Deployment

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx

ReplicaSet

apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx

Useful Commands

# See ownership chain
kubectl get deploy,rs,pod

# Show labels
kubectl get pods --show-labels

# Scale a Deployment
kubectl scale deployment nginx --replicas=5

# Watch a rollout
kubectl rollout status deployment/nginx

# See ReplicaSets created by a Deployment
kubectl get rs

# Explain fields during the exam
kubectl explain deployment.spec.selector
kubectl explain replicaset.spec.selector
kubectl explain pod.spec

Imperative Quick Commands for CKA Exam

# Pod
kubectl run nginx --image=nginx
kubectl run nginx --image=nginx --dry-run=client -o yaml > pod.yaml

# Deployment
kubectl create deployment nginx --image=nginx --replicas=3
kubectl create deployment nginx --image=nginx --dry-run=client -o yaml > deploy.yaml

# Scale and rollout
kubectl scale deployment nginx --replicas=5
kubectl rollout status deployment/nginx
kubectl rollout undo deployment/nginx

# Service for a Deployment
kubectl expose deployment nginx --port=80 --target-port=80 --name=nginx-svc
