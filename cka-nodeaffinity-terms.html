nodeAffinity — Terms vs Expressions

Understanding AND/OR logic in node selection rules.

The one line you need to remember

Terms are OR-ed.  |  Expressions inside a term are AND-ed.

The Structure, Layer by Layer

affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:           # LIST — items here are OR'd
        - matchExpressions:        # LIST — items here are AND'd
            - key: color
              operator: In
              values:
                - blue

Each level adds one piece of meaning:

  1. affinity → "scheduling rules ahead"
  2. nodeAffinity → "...about NODES specifically"
  3. requiredDuring... → "...and it's a hard rule"
  4. nodeSelectorTerms → "...here are my OR-groups"
  5. matchExpressions → "...each group has these AND-conditions"

The Recipe Analogy

a term = one complete recipe

Each term is a full recipe for an acceptable node. Each expression inside a term is an ingredient that recipe requires.

  • Multiple ingredients in one recipe → all must be present (AND)
  • Multiple recipes → any one of them works (OR)

Side-by-Side: One Term vs Two Terms

1 term, 2 expressions = AND

nodeSelectorTerms:
  - matchExpressions:
      - key: hardware
        operator: In
        values: [gpu]
      - key: zone
        operator: In
        values: [us-east]

Reads as:

hardware=gpu AND zone=us-east

Single recipe, two required ingredients.

2 terms = OR

nodeSelectorTerms:
  - matchExpressions:
      - key: hardware
        operator: In
        values: [gpu]
  - matchExpressions:
      - key: zone
        operator: In
        values: [us-east]

Reads as:

hardware=gpu OR zone=us-east

Two separate recipes, either works.

Full Example — Mixed AND/OR

Goal: Schedule on nodes that are (color=blue AND zone=us-east) OR (color=red)

apiVersion: v1
kind: Pod
metadata:
  name: flexible-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:           # Term 1
              - key: color
                operator: In
                values:
                  - blue
              - key: zone
                operator: In
                values:
                  - us-east
          - matchExpressions:           # Term 2
              - key: color
                operator: In
                values:
                  - red
  containers:
    - name: nginx
      image: nginx

Visual breakdown

nodeSelectorTerms:
  - matchExpressions:
      - color In [blue]    ┐ AND
      - zone  In [us-east] ┘
                                  ← OR ←
  - matchExpressions:
      - color In [red]

Truth Table

Node labelsMatches?Why
color=blue, zone=us-east YES Satisfies Term 1 fully
color=blue, zone=eu-west NO Term 1 fails (zone wrong); Term 2 fails (color not red)
color=red, zone=eu-west YES Satisfies Term 2 (color=red is enough)
color=red, zone=us-east YES Satisfies Term 2
color=green NO Neither term matches

What Counts as "One Term"?

A term is one item under nodeSelectorTerms — everything from one - to the next -.

nodeSelectorTerms:
  - matchExpressions:           # ─── TERM 1 starts ───
      - key: hardware
        operator: In
        values: [gpu]
                                # ─── TERM 1 ends ─────
  - matchExpressions:           # ─── TERM 2 starts ───
      - key: memory-tier
        operator: In
        values: [high]
                                # ─── TERM 2 ends ─────

The - at the term level is the boundary. Each list item under nodeSelectorTerms is one term, and each term is one self-contained recipe.

Why Not Just Use nodeSelector?

nodeSelector can only express AND between conditions. The moment you need OR logic, you've outgrown nodeSelector and have to reach for nodeAffinity. That's the main reason the heavier syntax exists.

NeedUse
Match all listed labels (AND)nodeSelector — simpler, 3 lines
OR between groupsnodeAffinity with multiple terms
NotIn / Exists / DoesNotExistnodeAffinity
Numeric comparisons (Gt, Lt)nodeAffinity
Soft preference (best-effort)preferredDuringScheduling...
✗ The Silent Killer — The indentation between "1 term with 2 expressions" and "2 terms with 1 expression each" looks almost identical, but the logic flips completely (AND ↔ OR).
In the exam, slow down at this exact spot. Count the - at the nodeSelectorTerms level — that's how many recipes you have.

Imperative Quick Commands for CKA Exam

There is no clean flag for nodeAffinity. Generate YAML, then add the affinity block under spec.affinity for a Pod or spec.template.spec.affinity for a Deployment.

# Generate a Pod YAML and edit affinity
kubectl run nginx --image=nginx --dry-run=client -o yaml > pod-affinity.yaml
vi pod-affinity.yaml
kubectl apply -f pod-affinity.yaml

# Generate Deployment YAML and edit template affinity
kubectl create deployment nginx --image=nginx --dry-run=client -o yaml > deploy-affinity.yaml
vi deploy-affinity.yaml
kubectl apply -f deploy-affinity.yaml

# Verify scheduling
kubectl get pods -o wide
kubectl describe pod <pod-name>
