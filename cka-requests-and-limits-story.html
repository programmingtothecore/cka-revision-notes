Requests & Limits — The Kitchen Story

Every permutation of CPU and memory requests/limits, told through chefs in a shared kitchen. With syntax, gotchas, and exam-grade rules.

Setting the Scene

Picture a shared restaurant kitchen. The kitchen is your Node. The chefs are your Pods. They share two finite things:

The Cast

  • CPU = the stove burners. Compressible. If a chef wants more, the others can take turns. Nobody dies, they just cook slower.
  • Memory = the countertop space. Incompressible. If you've laid out ingredients on it, you can't "share" it. If it's full and someone needs more, somebody gets thrown out of the kitchen.
  • request = "I need at least this much reserved before I'll even start." Used by the scheduler to pick a node.
  • limit = "I will never use more than this." Enforced at runtime by the kubelet via cgroups.
One sentence to remember: request is a promise the scheduler makes to your pod. limit is a leash the kubelet puts on your pod.

The Four Scenarios (per resource)

For each of CPU and memory, you have four ways to configure a container:

requestlimitWhat happensQoS
——Free-for-all. First to be killed.BestEffort

Story Mode: Five Chefs Walk Into a Kitchen

scenario 1 · neither set

Chef Carla — "Just let me cook"

Carla shows up with no booking. She'll grab whatever burner is free and spread her ingredients wherever there's space. Sounds chill — until the kitchen gets busy.

  • The scheduler will place her almost anywhere; it doesn't know what she needs.
  • If memory pressure hits the node, the kubelet evicts pods BestEffort first. Carla goes first.
  • She can also accidentally starve a more important pod by eating all the free CPU.

When it's OK: dev/test, throwaway jobs, demos. Never in prod.

# No resources block at all
spec:
  containers:
    - name: app
      image: nginx
scenario 2 · request only

Chef Ravi — "Reserve me a burner, I'll figure out the rest"

Ravi books one burner up front. The scheduler guarantees he gets it. But because there's no limit, on a quiet night he'll spread across every free burner.

  • Pros: guaranteed minimum, can burst when the node has spare capacity. Great for spiky workloads.
  • Cons (CPU): noisy-neighbor risk. He may hog CPU until another pod requests it.
  • Cons (memory): dangerous. He can grow until the node runs out of memory, triggering evictions or even OOMKilled on neighbours. The node itself can become NotReady.

When it's OK: CPU-only request for bursty workloads on under-utilised nodes. Avoid for memory in shared clusters.

resources:
  requests:
    cpu: "250m"
    memory: "256Mi"
  # no limits
scenario 3 · limit only

Chef Mei — "I won't take more than this"

Mei sets a ceiling but doesn't book. Kubernetes does her a favour: it auto-copies the limit into the request. So she's effectively in scenario 5 (request = limit).

  • Looks like the most restrictive case but it's actually the safest — she gets exactly what she asked for, no more, no less.
  • Resulting QoS is often Guaranteed (if both CPU and memory have only limits, both copied = both equal).
resources:
  limits:
    cpu: "500m"
    memory: "512Mi"
# scheduler treats requests as 500m / 512Mi too
scenario 4 · request < limit

Chef Diego — "Reserve one burner, I might use up to three"

Diego books a minimum and accepts a ceiling. This is the most common production pattern.

  • Scheduler reserves the request; kubelet enforces the limit.
  • Burstable: he can grow when the node is quiet, gets throttled / killed at the limit.
  • Tuning rule of thumb: request ≈ p50 usage, limit ≈ p99 + buffer.
resources:
  requests:
    cpu: "200m"
    memory: "256Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
scenario 5 · request = limit

Chef Priya — "Lock it in"

Priya books exactly what she'll use. No bursting, no surprises.

  • QoS = Guaranteed (only when both CPU and memory have request = limit for every container).
  • Last to be evicted under pressure. Predictable performance.
  • Lower utilisation — you pay for capacity you might not use.

When to use: latency-sensitive workloads, databases, anything that hates being throttled.

resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"

The Critical Difference: CPU vs Memory at the Limit

This is the one thing the exam loves to probe. What happens when you hit the limit depends on which resource you're talking about.

CPU — compressible

Hits limit → container gets throttled. Slowed down, but stays alive. No restart. You'll see high CPU throttling metrics.

Memory — incompressible

Hits limit → container is OOMKilled. Process killed, container restarts (per restartPolicy). Pod's RESTARTS count goes up.

Mnemonic: "CPU slows, memory dies." You can throttle a stovetop. You can't throttle a countertop — you have to clear it.

Where It Goes in the YAML

Resources live per container, inside spec.containers[]. Not on the pod, not on the spec. Each container in a pod has its own block.

apiVersion: v1
kind: Pod
metadata:
  name: web
spec:
  containers:
    - name: app
      image: nginx:1.27
      resources:                    # <— here, indented under the container
        requests:
          cpu: "250m"
          memory: "256Mi"
        limits:
          cpu: "500m"
          memory: "512Mi"
    - name: sidecar                 # second container = its own resources block
      image: busybox
      resources:
        requests:
          cpu: "50m"
          memory: "64Mi"

Inside a Deployment, it's nested deeper: spec.template.spec.containers[].resources.

Units — Don't Mix Them Up

ResourceValidMeans
CPU1, 0.5, 500m1 vCPU, half a vCPU, 500 millicores (= 0.5)
Memory (binary)128Mi, 1Gi, 2TiPowers of 1024. Use these.
Memory (decimal)128M, 1GPowers of 1000. Slightly less than Mi/Gi.

Common Syntax Mistakes

✗ Mistake 1 — Writing memory as 500m thinking it's "500 mega".
That's 500 milli-bytes ≈ 0.5 bytes. Container will fail to start.
✓ Use: 500Mi or 500M.
✗ Mistake 2 — Putting resources: at the pod level. 
# WRONG
spec:
  resources:          # ← invalid
    limits: ...
  containers:
    - name: app
✓ Always under each container.
✗ Mistake 3 — Setting limit lower than request.
API server rejects it: limits.cpu < requests.cpu. Same for memory.
✗ Mistake 4 — Quoting numbers wrong. Both cpu: 500m and cpu: "500m" work, but cpu: 0.5 (no quotes) is fine while memory: 256Mi often needs quotes — YAML can mis-parse the suffix. When in doubt, quote everything.
✓ Tip — Use kubectl run to scaffold quickly: 
kubectl run nginx --image=nginx --dry-run=client -o yaml \
  --requests='cpu=200m,memory=256Mi' \
  --limits='cpu=500m,memory=512Mi'
Then edit the YAML. Faster than typing the resources block from memory.

QoS Classes — Decided Automatically

You don't pick the QoS class; Kubernetes derives it from your requests/limits. It controls the eviction order under node pressure.

ClassConditionEviction Order
GuaranteedEvery container has request = limit for both CPU and memoryLast
BurstableAt least one request set, but not GuaranteedMiddle
BestEffortNo requests or limits anywhereFirst

Decision Cheat Sheet

  • Latency-sensitive / databases: request = limit (Guaranteed).
  • Bursty web app: request < limit (Burstable). Set memory request close to typical usage; CPU request lower.
  • Batch / CI runners: request only on CPU is fine; always set a memory limit to prevent runaway jobs.
  • Never in prod: nothing set anywhere.
  • Memory limit: almost always set. The cost of OOMKill is usually less than the cost of taking down a node.
  • CPU limit: controversial — many SREs omit CPU limits to avoid throttling latency. Set a CPU request always though.

Inspecting What's Set

# See the spec's resources block
kubectl get pod <name> -o jsonpath='{.spec.containers[*].resources}'

# Pretty per container
kubectl describe pod <name> | grep -A4 -E 'Limits|Requests'

# QoS class (derived)
kubectl get pod <name> -o jsonpath='{.status.qosClass}'

# Live usage (needs metrics-server)
kubectl top pod <name> --containers

Imperative Quick Commands for CKA Exam

# Pod with requests and limits
kubectl run nginx --image=nginx \
  --requests='cpu=100m,memory=128Mi' \
  --limits='cpu=500m,memory=256Mi'

# Generate YAML for a Deployment, then add resources under containers
kubectl create deployment nginx --image=nginx --dry-run=client -o yaml > deploy-resources.yaml
vi deploy-resources.yaml
kubectl apply -f deploy-resources.yaml

# Verify resources and QoS class
kubectl describe pod <pod-name> | grep -A4 -E 'Limits|Requests'
kubectl get pod <pod-name> -o jsonpath='{.status.qosClass}{"\n"}'
Final rule of thumb: always set a memory limit, always set a CPU request, and only set both equal when you truly need predictable performance.
